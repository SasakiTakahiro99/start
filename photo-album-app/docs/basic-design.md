## 1. 目的・要件の要約

### 1.1 このアプリの最重要ミッション
「どれもいい写真すぎて選べない」というユーザーの本質課題に対し、**選ぶ負担を極限まで減らす**こと。
心臓部は次の一連ループ（本MVPで通しでエンドツーエンド動作させる）:

```
探す → 提案（オススメ順3枚） → 選ぶ（または「全部入れる」で逃げる） → アルバムにまとめる → 共有
```

### 1.2 確定済みの前提方針（ユーザー合意済み）
1. プラットフォーム: Web アプリ
2. ゴール: 動く MVP。心臓部ループの成立を最優先し、UI は作り込みすぎない
3. データ/共有: ローカル/自前サーバ完結。共有はアカウント不要のトークン付きURLを自前発行。クラウドストレージ・外部アカウント認証は使わない
4. AI: ローカル完結・費用ゼロ・オフライン動作。クラウドAI APIへ画像を送信しない

### 1.3 スコープ（要件2章準拠）
| 優先度 | 内容 | 本MVPでの扱い |
| --- | --- | --- |
| MUST | 本のように閲覧・共有できるフォトアルバムを作れる | 対応（本書の中心） |
| WANT | 紙のフォトブック出力（印刷意識の設計） | 非対応。ただしデータ設計で解像度・縦横比を保持（将来拡張の布石のみ） |
| LATER | 色ごとのフォルダ分け | 非対応。主要色をメタ情報として保持し流用可能にしておく |
| LATER | 写真付き日めくりカレンダー | 非対応。データ・設計上の布石も置かない |

## 2. システム全体構成

### 2.1 構成図（責務）

```
┌───────────────────────────────────────────────────────────┐
│ フロントエンド（ブラウザ / SPA）                             │

│  │ 取り込みAPI │ 検索/提案API │ アルバム/共有API      │  │

│  │  - EXIF抽出                  │   │ 管理           │   │
│  │  - 主要色抽出                │   └─────┬──────────┘   │
│  │  - ブレ/明るさ算出           │         │              │
│  │  - CLIP埋め込み/ゼロショット │         │              │
│  │    タグ（フォールバック付き）│         │              │
│  └─────┬────────────────────────┘         │              │

         │                                  │
┌────────▼──────────────┐        ┌──────────▼──────────────┐
```

### 2.2 各モジュールの責務
- **フロントエンド**: 心臓部ループのUI提供。API呼び出しと表示のみに責務を限定。画像処理・スコアリングは行わない。
- **バックエンド(FastAPI)**: HTTP API提供、取り込み処理のオーケストレーション、検索/提案スコアリング、アルバム永続化、共有トークン発行/検証。
- **メタ情報付与パイプライン**: 取り込み時に写真1枚ごとにメタ情報（撮影日/場所/主要色/被写体タグ/ブレ・明るさ/CLIP埋め込み）を算出しDBへ保存。
- **SQLite**: メタ情報・埋め込み・アルバム・ページ配置・共有トークンの永続化。
- **ローカルファイルストレージ**: 原本画像とサムネイル/表示用リサイズ画像を保存。DBにはパスのみ保持。

### 2.3 デプロイ形態
- 単一マシン上でバックエンド(FastAPI/uvicorn)を `127.0.0.1` で起動。フロントは同一オリジンで配信（既存 pdf-edit-tool と同様、静的HTML/JSをFastAPIから配信する構成を許容）。
- 共有リンクは同一サーバがLAN内/ローカルで解決する読み取り専用ビューを返す（外部公開は本MVPのスコープ外。方式は6章参照）。

## 3. 技術スタックの選定と根拠

| 層 | 採用 | 根拠 |
| --- | --- | --- |
| バックエンド | Python 3.12 + FastAPI（uvicorn） | 推奨たたき台どおり。既存 pdf-edit-tool でも FastAPI 採用実績があり、環境（Python 3.12.10）と整合。AI/画像ライブラリ（PyTorch系・OpenCV・Pillow）がPythonに集中しており、AI処理と同一プロセスに置けるのが最大の利点。 |
| DB | SQLite | 推奨どおり。ローカル完結・ゼロ設定・単一ファイル。MVPの規模（個人利用・単一マシン）に十分。埋め込みベクトルはBLOB/JSONで保持し、件数規模的に全件コサイン類似度計算で足りる（専用ベクタDBは過剰）。 |
| 画像埋め込み/ゼロショットタグ | open_clip（PyTorch, CLIP ViT-B/32 相当） | 推奨たたき台を採用。キーワード（テキスト）と写真（画像）を同一ベクトル空間に埋め込み、コサイン類似度で「一致度」を算出できる。被写体タグも候補ラベル群との類似度でゼロショット付与でき、要件3.0/3.1/4.1/4.2を単一モデルで満たせる。**懸念（初回モデルDL・重い依存）にはフォールバック（8章）で対処**。 |
| EXIF抽出 | Pillow（＋必要に応じ piexif 等） | 撮影日・GPSをローカルで抽出。外部API不要。 |
| 主要色抽出 | Pillow + 単純量子化/k-means（軽量） | 外部API不要。色フォルダ分け(LATER)へ流用可能な形で保持。 |
| ブレ/明るさ算出 | OpenCV（ラプラシアン分散＝ブレ、輝度平均＝明るさ） | 要件4.2の「ブレ/明るさの基本チェック」を機械的に算出。軽量。 |
| フロントエンド | 軽量SPA。**Reactを基本とするが、MVP規模では素のHTML/JS(Vanilla)構成も許容** | 推奨はReact。ただし本MVPは画面数が限定的（6画面程度）で、既存 pdf-edit-tool は素のHTML/JSで成立している。見開きプレビュー/検索モーダル/期間UIはVanillaでも実装可能。**採用の最終判断は詳細設計に委ねる**が、外部クラウド依存を増やさない・セットアップを軽くする観点から、重いビルドチェーンを避けた構成を推奨（本書ではフレームワーク非依存の画面仕様として記述）。 |

### 3.1 推奨たたき台の妥当性検証（結論）
- FastAPI + SQLite + open_clip + OpenCV/Pillow の組み合わせは、要件（ローカル完結・費用ゼロ・オフライン・クラウド非送信）をすべて満たし、単一Pythonプロセスにまとめられるため運用が単純。**推奨たたき台を概ねそのまま採用**する。
- 変更点は1つ: フロントを「Reactに固定」ではなく「フレームワーク非依存の画面仕様」とし、MVPの軽量セットアップを優先（既存プロジェクトの実装パターンとも整合）。
- ベクタDBは導入しない（件数規模から不要。SQLite内保持＋全件類似度計算で十分）。

## 4. データモデル

論理モデル。物理カラム名・型・インデックスは詳細設計で確定。

### 4.1 エンティティ一覧
| エンティティ | 役割 |
| --- | --- |
| Photo | 取り込んだ写真1枚。原本/サムネイルのパス、基本メタ、品質指標を保持 |
| PhotoEmbedding | 写真のCLIP画像埋め込みベクトル（Photoと1:1） |

| ShareLink | アルバム共有用のトークン付きリンク |

### 4.2 主要項目

**Photo**
| 項目 | 内容 |
| --- | --- |
| id | 主キー |
| original_path | 原本画像ファイルへのパス |
| thumbnail_path | 表示用縮小画像へのパス |

### 4.3 将来拡張への配慮（データ設計での布石）
- **印刷解像度WANT**: Photo に `width/height/aspect_ratio` を保持。原本を無劣化で残し、表示はサムネイル利用。将来の印刷出力時に実解像度・縦横比を参照可能。
- **色フォルダ分けLATER**: Photo の `dominant_colors` を構造化して保持。色でのグルーピングクエリを後から追加できる。

## 6. 主要API一覧

エンドポイントは概略。認証はMVPでは共有ビューのトークン検証のみ（編集系APIはローカル前提で認証なし。将来のTODOとして明記）。

| メソッド/パス | 役割 | 入力（概要） | 出力（概要） |
| --- | --- | --- | --- |
| POST `/photos/import` | 写真取り込み＋メタ付与を起動 | 画像ファイル（複数可） | 取り込んだPhotoのidと処理ステータス |
| GET `/photos/{id}/status` | メタ付与の進捗確認 | photo_id | meta_status、フォールバック適用有無 |
| GET `/photos/{id}/thumbnail` | サムネイル画像取得 | photo_id | 画像バイナリ |
| POST `/search/keyword` | キーワード検索→オススメ順候補 | keyword, 件数(既定3), （任意）albumコンテキスト | 候補写真リスト（各: photo_id, オススメ順位, 一致度, 品質, サムネイルURL） |
| GET `/search/period` | 期間選択→候補 | granularity(day/month), 期間指定（任意=空欄可） | 候補写真リスト（オススメ順3枚） |
| GET `/search/period/suggest` | 空欄時の期間提案（「この月はどう？」） | なし/ヒント | おすすめ期間（例: 写真が多い月）とプレビュー |
| POST `/albums` | アルバム作成 | title | album_id |
| POST `/albums/{id}/photos` | 選んだ写真をアルバムに追加＋自動配置 | photo_idの配列（1枚 or 3枚一括） | 更新後のページ構成 |
| GET `/albums/{id}/pages` | 見開きプレビュー用ページ構成取得 | album_id | ページ配列（page_index, layout, 配置写真） |
| POST `/albums/{id}/share` | 共有リンク発行 | album_id | token付きURL |
| DELETE `/albums/{id}/share/{token}` | 共有リンク無効化 | token | 結果 |
| GET `/share/{token}` | 共有ビュー用データ取得（読み取り専用） | token | アルバムのページ構成（編集不可） |

### 6.1 検索候補レスポンスの必須項目（要件3.1/3.2）
- 候補は既定3枚。各候補に **オススメ順位（1,2,3）** と、その根拠となる **一致度** を含める（要件「各候補にオススメ順位を表示」）。
- レスポンスとは別に、フロントは「3枚まとめて全部入れる」操作で全候補のphoto_idをまとめて `POST /albums/{id}/photos` に渡す（逃げ道）。

## 7. メタ情報自動付与の処理方針（要件3.0 / 4.1 / 4.2）

取り込み時、写真1枚ごとに以下を算出しDBへ保存する。重い処理はバックグラウンドで進め、S1で進捗表示（手を止めさせない）。

| 項目 | 手段 | 出力 | フォールバック時 |
| --- | --- | --- | --- |
| 撮影日時 | EXIF（Pillow） | taken_at | EXIF無ければファイル更新日時 |
| 場所 | EXIF GPS | gps_lat/lng | 無ければ null（許容） |
| 主要色 | Pillow＋量子化/簡易k-means | dominant_colors | モデル非依存のため常時算出可 |
| ブレ | OpenCV ラプラシアン分散 | blur_score | モデル非依存のため常時算出可 |
| 明るさ | OpenCV 輝度平均 | brightness_score | モデル非依存のため常時算出可 |
| 画像埋め込み | open_clip 画像エンコーダ | PhotoEmbedding.vector | モデル未取得時は生成しない（8章のフォールバック検索へ） |
| 被写体タグ | open_clip でゼロショット（候補ラベル群との類似度上位を付与） | PhotoTag（label, score） | モデル未取得時は色/明るさから機械的な簡易タグ（例: 暗い/明るい/暖色/寒色）で最低限を付与 |

- 候補ラベル群（海/食べ物/夜景/人物 等）は設定として持ち、拡張可能にする（要件4.1「被写体タグ中心から開始」）。
- 抽象語（笑顔/エモい等）は初期はCLIPのテキスト一致に委ね、精度は許容範囲で段階拡張（要件4.1「段階的に拡張」）。

## 8. オススメ順スコアリングの方針（要件4.2）

初期実装は **「一致度 ＋ ブレ/明るさの基本チェック」**。好み学習（過去の選択傾向）は本MVPでは実装しない（項目・拡張余地のみ残す）。

- **一致度（match_score）**:
  - キーワード検索: キーワードのCLIPテキスト埋め込みと各写真の画像埋め込みのコサイン類似度。
  - 期間選択: 期間条件に合致する写真を母集団とし、一致度は品質中心（キーワードが無いため）。
- **品質（quality_score）**: blur_score・brightness_score を正規化して合成（ブレすぎ/暗すぎ/明るすぎにペナルティ）。構図は本MVP対象外。
- **総合スコア**: `total = w1 * match_score + w2 * quality_score`（重みは設定値。詳細設計で初期値決定）。上位3枚を候補として、その順位を「オススメ順位」として返す。
- フォールバック時（埋め込みなし）: 一致度は被写体タグ一致・色/日付一致などのメタ照合で代替し、品質チェックは通常どおり適用（心臓部ループは成立させる）。

## 9. 共有リンク（トークン付きURL）の方式（要件3.4）

- アルバム共有時、`ShareLink` に **推測困難なランダムトークン**（十分な長さの安全な乱数）を発行し、`/share/{token}` 形式のURLを返す。
- `GET /share/{token}` はトークンを検証し、対応アルバムの **読み取り専用** ページ構成を返す。編集系APIは共有ビューから呼べない。
- **アカウント不要**: トークンを知っていれば誰でも閲覧可（要件の中核）。認証・ログインは要求しない。
- **停止手段**: `revoked` フラグ／`DELETE .../share/{token}` で無効化可能。`expires_at` 項目を用意（MVPでは無期限許容）。
- MVPではローカル/LAN内での解決を前提とし、インターネット公開（外部からの到達性確保）はスコープ外。ただしトークン方式はそのまま外部公開時にも流用できる設計とする。

## 10. モデル未取得時のフォールバック方針

CLIPモデルのダウンロード・重い依存が未整備でも、心臓部ループをエンドツーエンドで成立させるため、段階的フォールバックを設ける。

| 状態 | 挙動 |
| --- | --- |
| モデル利用可 | CLIP埋め込み・ゼロショットタグをフル利用。キーワード検索は意味的類似で高精度。 |
| モデル未取得/ロード失敗 | 埋め込み生成をスキップ。検索は「被写体タグ（簡易タグ）＋日付/場所/主要色」のメタ照合で代替。品質チェック（ブレ/明るさ）は常時有効。逃げ道ボタン・期間導線・見開き・共有はすべて通常動作。 |

- どちらの状態でも API 形状・画面遷移は同一（フロントはフォールバックを意識しない）。
- メタ情報のうち **EXIF・主要色・ブレ/明るさは常にモデル非依存で算出可能**なため、最低限の検索と品質判定は常に成立する。
- セットアップ手順にモデル取得方法と「初回モデルロードが重い」旨を明記する前提（制約は11章）。

## 11. 制約事項・非対応範囲

- **非対応（本MVPスコープ外）**: 紙フォトブック出力(WANT)、色フォルダ分け(LATER)、日めくりカレンダー(LATER)、好み学習によるオススメ順の高度化、構図評価、インターネット越しの外部公開共有。
- **制約**:
  - クラウドAI API・クラウドストレージ・外部アカウント認証は使用しない（要件・合意方針）。
  - CLIPモデルは初回にダウンロードが必要で、初回ロードが重い（セットアップ手順に明記）。
  - 埋め込み検索は全件コサイン類似度計算（写真枚数が非常に多いと遅くなりうる。MVP規模では許容）。
  - GPS/EXIFを持たない写真は場所/撮影日の一部が欠落しうる（フォールバックで最低限動作）。
  - 編集系APIはローカル利用前提で認証なし（外部公開時に要追加。TODOとして明記）。

| 要件（MUST） | 対応する設計項目 | 充足 |
| --- | --- | --- |

| 3.2 候補から1枚ずつ選べる ＋「3枚まとめて全部入れる」逃げ道ボタン | S3/S4のUI仕様 ＋ `POST /albums/{id}/photos`（1枚/一括両対応） | ○ |
