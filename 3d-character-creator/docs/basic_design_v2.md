# 3Dキャラクタークリエイター リニューアル版（AI 3Dモデル生成方式） 基本設計書 v2

本書は `docs/basic_design.md`（v1・MVP版）を置き換えるものではなく、方針転換に伴う
**リニューアル版**として新規作成する。v1は「プリミティブ形状の組み合わせによるMii風スタイライズド
キャラクター生成」を採用していたが、v2では「AI 3Dモデル生成APIによるリアルな人物3Dモデルの
都度生成」に全面転換する。v1からの変更点は都度「v1からの変更点」として明記する。

## 0. v1からの変更点（サマリ）

| 観点 | v1（MVP・プリミティブ方式） | v2（本書・AI生成方式） |
|---|---|---|
| キャラクター生成方式 | 球・カプセル等プリミティブの組み合わせ＋パラメトリック変形 | AI 3Dモデル生成API（Meshy AI想定）によるGLB生成 |
| 見た目のテイスト | Mii/VRChat簡易アバター的なスタイライズド頭身 | リアルな人物を志向したフォトリアル寄りの3Dモデル |
| カスタマイズUX | スライダー操作でリアルタイム連続変形 | 性別・雰囲気・体型傾向・容姿テキスト等を入力→生成ボタン→非同期生成（数十秒〜数分）→結果表示 |
| 外部アセット | 一切使用しない（プリミティブのみ） | AI生成APIが返すGLBファイルを外部アセットとして使用する |
| サーバーサイド | 新規実装なし（ブラウザのみで完結） | 薄いプロキシサーバー（Node.js/Express想定）を新設し、APIキーを保持してAI生成APIを中継する |
| 移動操作・カメラ・地面フィールド | WASD/矢印キー移動＋三人称追従カメラ＋簡易地面 | 同上の挙動を維持し、既存実装（field.js, controls.js）をほぼそのまま流用する |
| 保存対象（localStorage） | パーツごとの数値・色パラメータ一式 | 生成に使ったユーザー入力パラメータ＋生成結果のGLB参照情報（URL/ID等） |
| ビルド不要・サーバー実装なしの制約 | 制約あり（v1の非機能要件） | **撤廃**（秘書承認済み）。ただし新設は薄いプロキシに限定し過剰な作り込みは避ける |

## 1. 目的・要件の要約

ユーザーの「本当にリアルな人間のような、きれいなキャラクターを作りたい」という要望を受け、
既存MVP（プリミティブ形状によるスタイライズド生成）を、AI 3Dモデル生成API（Meshy AI または
Tripo AIを想定）を用いたリアルな人物モデルの都度生成方式へ全面リニューアルする。

### 要件サマリ
- ユーザーが性別・雰囲気・体型傾向・容姿のテキスト説明などを入力し、それをもとにプロンプトを
  組み立ててAI生成APIへ送信、リアルな人物の3DモデルをGLB形式で生成する。
- 生成されたGLBをThree.jsシーンにロードし、既存同様に地面フィールド上をWASD/矢印キーで
  歩行操作できるようにする（三人称追従カメラも維持）。
- 生成は非同期・数十秒〜数分かかるため、「入力→生成ボタン→ローディング表示→完了後表示」という
  非リアルタイムのUXに変更する。
- APIキーの漏洩を避けるため、ブラウザから直接AI生成APIを叩かず、薄いプロキシサーバー
  （Node.js想定）を新設してAPIキーをサーバー側のみで保持する。
- 実APIキーは未提供のため、環境変数から読み込む前提とし、未設定時はエラーメッセージ表示または
  デモ用のダミーGLB表示にフォールバックする。
- 生成結果の参照情報・生成時の入力パラメータは`localStorage`で保存・復元する。
- 既存の移動操作・カメラ・地面フィールドの実装は可能な限り再利用する。
- 本プロジェクト以外（リポジトリ直下のPDF編集ツール等）には一切手を加えない。

### 想定案件規模
既存MVPからの方式転換ではあるが、対象は依然として単一機能規模のミニアプリである。
過剰な作り込み（認証基盤・DB導入・フルWebフレームワーク構成・複数ユーザー管理等）は行わない。

## 2. 採用API・技術選定

### 2.1 AI 3Dモデル生成API: Meshy AIを主対象とする

比較検討の要点（Web検索によらない、一般的に知られている特徴・API形態からの想定判断。
**不確実な部分は詳細設計時に公式ドキュメントを要確認**とする）:

| 観点 | Meshy AI | Tripo AI |
|---|---|---|
| API提供形態 | REST API（Text-to-3D / Image-to-3D）を公式に提供している想定 | 同様にREST APIを提供している想定 |
| 出力形式 | GLB/FBX/OBJ等、GLBを含む複数形式に対応している想定 | GLBを含む複数形式に対応している想定 |
| 生成方式 | テキストプロンプトからのText-to-3D、画像からのImage-to-3Dの両方に対応している想定 | 同様にText-to-3D/Image-to-3Dに対応している想定 |
| 人物のリアルさ・実績の知名度 | 3Dアセット生成サービスとして比較的知名度が高く、情報・実装事例が見つけやすいと想定される | 同様に3D生成サービスとして知られるが、Meshyと比べ日本語情報量ではやや少ない想定 |
| 非同期ジョブ方式 | 生成リクエスト受付→ジョブID発行→ポーリングでステータス確認→完了後URL取得、という非同期方式を取ると想定される | 同様の非同期方式を取ると想定される |

判断: 両者ともAPI形態・出力形式（GLB対応）・非同期ジョブ方式という点で本アプリの要件を
満たせる想定であるため、**知名度・情報量の観点でMeshy AIを主対象**として設計を進める。
ただし本書の内容は「非同期ジョブ方式でGLBを取得する」という抽象度に留め、実際のエンドポイント名・
リクエスト/レスポンス形式・認証ヘッダ形式・ポーリング間隔の具体値は**詳細設計時に公式ドキュメントを
確認の上で確定**する。Tripo AIは代替候補として位置づけ、Meshy AI利用時に致命的な制約
（レート制限・料金体系・人物生成の品質不足等）が判明した場合の切替先候補として申し送る。

### 2.2 プロキシサーバー: Node.js/Express

- APIキーをブラウザに露出させないため、簡易なプロキシサーバーを新設する。
- 技術選定: **Node.js + Express**（軽量で薄い中継用途に適し、追加の学習コストが小さいため）。
- 役割: ブラウザからのモデル生成リクエストを受け、サーバー環境変数に保持したAPIキーを付与して
  Meshy AI（またはTripo AI）へリクエストを中継する。生成ジョブのステータス確認・完了後のGLB URL
  取得についても同様に中継する。
- 想定エンドポイント概要（詳細は詳細設計に譲る）:
  - `POST /api/generate` : ユーザー入力パラメータを受け取り、AI生成APIへText-to-3D生成ジョブを発行、
    ジョブIDを返す。
  - `GET /api/generate/:jobId/status` : 指定ジョブの生成状況（進行中/完了/失敗）とGLB URL
    （完了時）を返す。
  - APIキー未設定時は上記いずれも分かりやすいエラー（例: 503相当＋エラーメッセージ）を返すか、
    デモ用フォールバック（ダミーGLBのURLを返す）を行う。挙動の切り替えは詳細設計で確定する。
- サーバーの永続化（DB等）は行わない。生成ジョブの状態はプロキシサーバーがステートレスに
  AI生成API側へ都度問い合わせる、またはメモリ上の簡易キャッシュに留める程度とし、DB導入は行わない。
- 認証基盤・ユーザー管理は導入しない（単一ユーザー・ローカル/簡易ホスティング利用を想定）。

## 3. 既存ファイルの流用・改修方針一覧

| ファイル | 方針 | 理由・改修概要 |
|---|---|---|
| `js/character.js` | **作り直し** | プリミティブ形状構築ロジック（`createCharacter`/`updateCharacterPart`/`layoutParts`等）は方式転換により不要。GLBモデルのロード・シーンへの配置・破棄（モデル差し替え時のdispose）を担う新実装に置き換える。**ただし`controls.js`が引き続き操作対象とする`character.root`（`THREE.Group`。位置・回転を操作される）というインターフェースは、新実装でも必ず維持しなければならない（9章参照）** |
| `js/params.js` | **作り直し** | 数値スライダー用の値域・体型プリセット定義ではなく、生成パラメータ（性別・雰囲気・体型傾向・容姿テキスト等）の入力項目定義・バリデーション・デフォルト値を扱う内容に置き換える |
| `js/storage.js` | **改修** | `localStorage`キー・保存/読込という枠組み自体は流用するが、保存対象を「生成パラメータ＋生成結果のGLB参照情報」に変更する（5章参照）。関数シグネチャの骨格（`saveToStorage`/`loadFromStorage`/`clearStorage`）は踏襲可能 |
| `js/customization-ui.js` | **作り直し** | スライダー/カラーピッカーのリアルタイム反映UIから、テキスト入力・セレクト等の生成パラメータ入力フォーム＋「生成」ボタン＋ローディング表示の管理へ役割を変更する |
| `js/field.js` | **そのまま流用** | 地面フィールドの生成ロジックは生成方式と無関係のため変更不要 |
| `js/controls.js` | **そのまま流用（ただし正面向きの前提（+Z方向）が崩れる場合は補正処理の追加が必要。9章参照）** | WASD/矢印キー移動・三人称追従カメラは`character.root`（`THREE.Group`）の位置・回転を操作する設計であり、新`character.js`が`character.root`というインターフェースを維持する限り流用可能。ただし現行実装は`atan2(x, -z)`によりキャラクターの正面をローカル+Z方向と決め打ちしてコメントにも明記されており、AI生成モデルの正面向きが不定な場合はこの前提が崩れ、ロード時の一律回転などの補正処理を追加実装する必要がありうる。「ほぼノータッチでよい」ものではなく、9章の申し送り内容と同等の重みを持つ確認・改修対象として扱うこと |
| `js/main.js` | **改修** | 初期化・結線ロジックの骨格（Scene/Camera/Renderer生成、フィールド・キャラクター追加、アニメーションループ）は流用しつつ、キャラクター初期化を「即時プリミティブ構築」から「保存済みGLB参照があればロード、無ければ未生成状態で待機」という非同期フローに変更する。プロキシサーバーへのHTTP通信（生成リクエスト・ポーリング）の起点もmain.js側に追加する想定 |
| `index.html` | **改修** | カスタマイズパネル部分のコンテナ構造・ローディング表示用の要素追加が必要。`importmap`に`GLTFLoader`読み込み口を追加する |
| `css/style.css` | **改修** | ローディング表示・生成フォームのスタイルを追加する程度の改修 |

補足: 上記はあくまで基本設計の方針であり、具体的な関数シグネチャ・処理内容は詳細設計書で確定する。

## 4. 画面構成・UI概要

v1と同様に単一画面（1ページ）構成、「3Dキャンバス領域」と「カスタマイズパネル」の2ブロック分割は
維持する。パネルの中身をスライダー群からAI生成パラメータ入力フォームへ変更する。

```
+--------------------------------------------------------------+
| ヘッダー: タイトル / 保存状態表示                              |
+---------------------------------------+----------------------+
|                                       |  生成パラメータパネル  |
|                                       |  ------------------  |
|                                       |  [性別] セレクト       |
|         3Dキャンバス領域              |  [雰囲気] セレクト     |
|         （Three.jsレンダラー）         |         (例: 明るい/クール/ |
|         キャラクター(GLB) + 地面      |          かわいい 等)  |
|         フィールド                    |  [体型傾向] セレクト   |
|         三人称追従カメラ               |         (例: スリム/標準/  |
|                                       |          がっしり 等)  |
|                                       |  [容姿の説明] テキスト |
|                                       |         エリア        |
|                                       |  [生成]ボタン          |
|                                       |  ------------------  |
|                                       |  ローディング表示領域  |
|                                       |  （進捗メッセージ・    |
|                                       |    スピナー等）        |
|                                       |  ------------------  |
|                                       |  「保存」「リセット」  |
|                                       |  ボタン               |
+---------------------------------------+----------------------+
| フッター: 操作説明（WASD/矢印キーで移動 の案内テキスト）        |
+--------------------------------------------------------------+
```

### v1からのUI変更点
- スライダー・カラーピッカーによる「操作すると即座に反映される」プレビューUIは廃止する。
  （生成は非同期・数十秒〜数分かかるため、リアルタイム反映は実現不可能。）
- 「生成」ボタン押下後、生成完了までカスタマイズパネルはローディング状態を表示し、
  再度の生成リクエストを防ぐため「生成」ボタンは無効化する。
- 生成完了後、生成されたキャラクターがキャンバスに表示され、ローディング表示は消える。
- 「保存」ボタンは生成パラメータ＋生成結果のGLB参照情報を保存する（挙動は生成完了後のみ
  有効化するか等、詳細設計で確定）。
- 「リセット」ボタンはv1同様、表示・保存データの初期化を担う（詳細挙動は詳細設計で確定）。
- 生成失敗時（APIキー未設定・API側エラー・タイムアウト等）はエラーメッセージをパネル内に表示する。

## 5. 機能一覧

| # | 機能 | 概要 |
|---|------|------|
| 1 | 生成パラメータ入力 | 性別・雰囲気・体型傾向・容姿説明テキスト等をUIで入力する |
| 2 | AIモデル生成リクエスト | 「生成」ボタン押下時、入力パラメータからプロンプトを組み立て、プロキシサーバー経由でAI生成API（Meshy AI想定）へText-to-3D生成を依頼する |
| 3 | 生成進捗確認・ローディング表示 | 生成ジョブの完了をポーリング等で待ち、待機中はローディング表示を行う |
| 4 | 生成結果の表示 | 完了後に取得したGLBファイル（URL）を`GLTFLoader`でロードし、`character.root`配下に配置してシーンに表示する |
| 5 | フィールド表示（v1流用） | 小さめの平面地面をシーンに配置し、キャラクターが乗る土台とする |
| 6 | 移動操作（v1流用） | WASD/矢印キー入力でキャラクターを地面平面上で移動・向き変更させる。カメラは三人称追従を維持する |
| 7 | 保存/復元 | 生成パラメータおよび生成結果のGLB参照情報を`localStorage`に保存し、ページロード時に復元してモデルを再ロードする |
| 8 | フォールバック動作 | AI生成APIキー未設定時、分かりやすいエラー表示、またはデモ用ダミーGLBによる代替表示を行う |

対応範囲外（本リニューアルでは扱わない）:
- スライダー等によるリアルタイム連続変形（AI生成方式の性質上、実現不可能なため非対応）
- 生成済みモデルの部分的な後編集（顔パーツの微調整等）
- 複数キャラクターの管理・切り替え、キャラクターの共有・エクスポート機能
- ユーザーアカウント・認証・マルチプレイ
- モバイル/タッチ操作対応
- 実際のAPIキーを用いた本番的な生成動作確認（キー未提供のため、モック/スタブによる確認に留める。7章・8章参照）

## 6. 主要なデータ構造（概要）

### 6.1 生成パラメータ（AI生成APIへのプロンプト組み立て元）

```jsonc
{
  "version": 2,                    // v1のバージョン1から変更。データ構造の互換性なし
  "gender": "female",              // 性別（"female" | "male" | 未指定 等、詳細は詳細設計で確定）
  "mood": "bright",                // 雰囲気プリセット（例: 明るい/クール/かわいい 等）
  "bodyType": "slim",              // 体型傾向プリセット（例: スリム/標準/がっしり 等）
  "appearanceDescription": "..."   // 容姿の自由記述テキスト（プロンプトへ組み込む自然文）
}
```

### 6.2 生成結果参照情報

```jsonc
{
  "modelId": "job-xxxxxxxx",        // AI生成API側のジョブID/モデルID
  "modelUrl": "https://.../model.glb", // 取得済みGLBのURL（プロキシ経由で取得したもの）
  "generatedAt": "2026-07-06T12:34:56Z", // 生成完了日時
  "sourceParams": { /* 6.1のオブジェクト。生成時に使ったパラメータのスナップショット */ }
}
```

### 6.3 `localStorage`保存構造（概要）

`storage.js`が保存するトップレベルオブジェクトは、v1の「パラメータ一式」から
「生成パラメータ＋生成結果参照情報のペア」に変更する。

```jsonc
{
  "version": 2,
  "params": { /* 6.1の生成パラメータ（次回生成フォームの初期値として復元） */ },
  "generatedModel": { /* 6.2の生成結果参照情報。未生成の場合はnull */ }
}
```

- ページロード時、`generatedModel`が存在すれば`modelUrl`から`GLTFLoader`で再ロードして表示する。
  存在しない場合は未生成状態（生成前の空のキャンバス、またはプレースホルダー表示）とする。
- GLBの実体（バイナリ）自体は`localStorage`に保存しない（サイズ的に不適）。あくまでURL/IDなど
  再取得に必要な参照情報のみを保存する。
- **バージョン非互換時の基本方針**: `version`フィールドが2以外（v1の`1`を含む）の場合、既存の
  保存データとの互換性を取ろうとせず、既存データを無視して未生成状態（生成前の空のキャンバス）
  から開始することを基本方針とする。データ移行・変換ロジックは設けない。具体的な判定・破棄処理は
  詳細設計書で確定する。
- 具体的なキー名・フィールド名・バリデーション仕様は詳細設計書で確定する。

## 7. 外部インターフェース

- **AI 3Dモデル生成API（Meshy AI想定、代替候補Tripo AI）**: プロキシサーバー経由でのみ呼び出す
  （ブラウザから直接呼び出さない）。非同期ジョブ方式（生成リクエスト→ジョブID→ポーリング→
  完了時GLB URL取得）を想定。具体的なエンドポイント・リクエスト/レスポンス形式・認証方式は
  詳細設計時に公式ドキュメントを確認して確定する。
- **プロキシサーバー（Node.js/Express、本アプリ内に新設）**: ブラウザ（`main.js`）から
  `fetch`等でHTTP通信を行う。エンドポイント概要は2.2節の通り。APIキー（例: `MESHY_API_KEY`）は
  プロキシサーバーの環境変数として保持し、ブラウザには一切露出させない。
- **GLTFLoader（Three.js公式アドオン）**: プロキシ経由で取得したGLBのURLをロードし、
  `THREE.Group`としてシーンに追加する。`three/addons/loaders/GLTFLoader.js`をimport mapに追加する。
- **Three.js（CDN、v1から継続）**: 引き続きCDN経由・ビルド不要で読み込む。バージョン・URL等の
  詳細は詳細設計時に必要に応じ見直す（GLTFLoaderの動作要件に合わせる）。

## 8. 非機能要件・制約

- **APIキー管理**: AI生成APIのキーはブラウザに一切露出させず、プロキシサーバーの環境変数
  （例: `MESHY_API_KEY`）から読み込む。未設定時は分かりやすいエラーメッセージを返すか、
  デモ用プレースホルダー動作（ダミーGLB表示等）にフォールバックする。実際のキーを用いた
  本番的な動作確認は本リニューアルの対象外とし、モック/スタブによる確認に留める（後工程の
  テスト仕様書・実装にもこの前提を申し送る）。
- **既存プロジェクトとの独立性（v1から継続）**: 本アプリのファイルは全て`3d-character-creator/`
  配下に配置し、プロキシサーバーの新規実装も同ディレクトリ配下に収める。リポジトリルート直下の
  既存PDF編集ツール関連ファイル（`index.html`, `api_server.py`, `pdf_editor.py`等）には
  一切手を加えない・混在させない。
- **サーバー実装の範囲限定**: プロキシサーバーはAPIキー中継を目的とした薄い実装に限定する。
  フルWebフレームワーク構成・DB導入・認証基盤・セッション管理等の過剰な作り込みは行わない。
- **ビルド不要方針の扱い（v1からの変更）**: v1の「ビルド不要・サーバーサイド新規実装なし」制約は
  撤廃する（秘書承認済み）。ブラウザ側（フロントエンド）は引き続きビルドステップなしのCDN構成を
  維持する。サーバーサイド（プロキシ）のみNode.js実行環境を必要とする点が新たな前提となる。
- **対象ブラウザ（v1から継続）**: 最新版のChrome/Edge（Windows）を主対象とする。WebGL対応が前提。
- **パフォーマンス**: AI生成モデルはポリゴン数・テクスチャ解像度が生成条件により変動しうるため、
  極端に重いモデルが返るケースへの対処（表示の重さの許容範囲、必要ならモデル読み込み失敗時の
  フォールバック等）は詳細設計時に検討する。
- **国際化・アクセシビリティ（v1から継続）**: 画面文言は日本語のみを想定し、多言語対応は行わない。
- **著作権・利用規約への配慮**: AI生成APIの利用規約・生成物の権利関係は本書のスコープ外とするが、
  実運用に進める場合は別途確認が必要である旨を申し送る。

## 9. 後工程（詳細設計）への申し送り事項

- Meshy AI（および代替候補Tripo AI）の公式ドキュメントを確認し、以下を確定させること:
  - 実際のエンドポイントURL・認証ヘッダ形式・リクエスト/レスポンスのJSON構造
  - Text-to-3D生成ジョブの非同期フロー（ポーリング間隔・タイムアウト目安・エラーコード体系）
  - 出力GLBの取得方法（署名付きURLの有効期限等、`localStorage`保存内容への影響有無）
- プロキシサーバー（Node.js/Express）の具体的なディレクトリ構成・起動方法・環境変数名
  （例: `MESHY_API_KEY`）・エンドポイント仕様（リクエスト/レスポンススキーマ）を確定すること。
- APIキー未設定時のフォールバック仕様（エラー表示 or デモ用ダミーGLB表示）をどちらにするか、
  ダミーGLBを用意する場合の入手・配置方法を確定すること。
- プロンプト組み立てロジック（性別・雰囲気・体型傾向・容姿テキストからどのような自然文プロンプトを
  構成するか）の具体的なテンプレートを設計すること。あわせて、このロジックをどのモジュールに
  置くか（生成パラメータの定義を担う`params.js`に含めるか、独立した新規モジュールとして切り出すか）
  の役割分担を詳細設計側で決定すること。
- `js/controls.js`の正面向き判定（`atan2(x, -z)`により、キャラクターの正面をローカル+Z方向と
  決め打ちする実装になっている）が、AI生成モデルの正面方向と整合するか確認すること。この前提は
  `controls.js`をそのまま流用できるかどうかを左右する重要な確認事項であり、「ほぼノータッチでよい」
  類の軽微な確認ではない。モデルによって正面向きが不定な場合の補正方法（例: ロード時に
  `character.root`またはGLBモデル自体へ一律回転を加える等）を具体的に検討・実装すること。
- 新`character.js`は、`controls.js`が位置・回転を操作する対象として`character.root`
  （`THREE.Group`）を返すインターフェースを必ず維持すること。この契約が崩れると`controls.js`の
  そのまま流用が成立しないため、詳細設計・実装の双方でこの制約を前提とすること。
- 生成中・生成失敗時のUI状態遷移（ボタンの活性/非活性、エラーメッセージの文言、再試行導線）を
  具体化すること。
- `localStorage`のキー名・バージョン互換性（v1データが残っている環境でv2を開いた場合の扱い。
  例: バージョン不一致時はv1データを無視してデフォルト状態から開始する、等）を確定すること。
- テスト仕様書作成時は、実APIキーが無い前提でのモック/スタブテスト方針（プロキシサーバーの
  AI生成API呼び出し部分をモック化する等）を前提とすること。
