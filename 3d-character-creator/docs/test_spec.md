# 3Dキャラクタークリエイター MVP テスト仕様書

## 0. 本書の位置づけ

- 対象: `3d-character-creator/` 配下の実装一式（`index.html`, `js/*.js`, `css/style.css`）
- 参照元: `docs/basic_design.md`（承認済み基本設計書）, `docs/detail_design.md`（4回のレビュー修正・実装後コードレビュー指摘対応まで反映済みの詳細設計書）
- 実装コード（`character.js`, `controls.js`, `customization-ui.js`, `main.js`, `params.js`, `storage.js`, `field.js`, `index.html`, `style.css`）を実際に読み込んだ上でテストケースを作成している。
- 既存の自動テストコード（`test_pdf_editor.py`）は別プロジェクト（PDF編集ツール）用であり、本アプリ向けの既存テストは存在しない。そのため本書が本アプリの最初のテスト仕様書となる。
- 本アプリはビルド不要のブラウザ完結型SPAであり、UIとThree.jsの描画結果の確認が中心となるため、**手動でのブラウザ確認が主体**。一部はブラウザの開発者コンソール（DevTools Console）からJSを直接呼び出すことで自動化・効率化できるため、該当ケースには「実施方法」に代替コンソール手順を併記する。

### 0.1 テスト前提条件・環境

- ブラウザ: 最新のGoogle Chrome または Microsoft Edge（Windows）。基本設計書7章の対象範囲に準拠。
- 起動方法: `file://` 直開きは `type="module"` のCORS制約で失敗する可能性があるため、`3d-character-creator/` をカレントディレクトリとして `python -m http.server` 等の簡易サーバーを起動し、`http://localhost:8000/index.html` にアクセスする。
- 各テストケース実施前は、特記が無い限り以下の状態でクリーンに開始する。
  1. DevTools Console で `localStorage.removeItem('3d-character-creator:params')` を実行、またはブラウザのサイトデータを削除
  2. ページをリロード
- 本アプリの `localStorage` キーは `3d-character-creator:params` のみ（storage.js 8.1節）。

### 0.2 用語

- 「反映される」= 3Dキャンバス上のキャラクターの見た目（形状・色・位置）が操作直後に変化すること（リアルタイムプレビュー）。
- 「表示値」= カスタマイズパネル上のスライダー・カラーピッカー・セレクトのUI表示値（`input.value`）。
- `changedPath` = 詳細設計7.3節・4.4.2節で定義される、各コントロールが担当するドット区切りのパラメータパス。

## 2. テストケース一覧

凡例: 重要度 ★★★=重点確認項目（開発経緯で指摘された修正箇所）、★★=通常、★=軽微

### A. 初期表示

#### A-1 初期表示（WebGL対応環境）★★

- 目的: アプリ起動直後に必要な要素が全て表示されることを確認する。
- 手順:
  1. `localStorage` をクリアした状態でページを開く。
  2. 画面全体を目視確認する。
- 期待結果:
  - ヘッダーに「3Dキャラクタークリエイター」というタイトルと保存状態表示（`#save-status`）が表示される。
  - 保存状態表示の初期値は「デフォルト設定で開始しました」（`restored:false` のケース。main.js 10.3節）。
  - 3Dキャンバス領域（`#canvas-container`）に空色の背景（`0x87ceeb`）、緑色の地面（`0x6fae5c`）とグリッド線、デフォルトパラメータのキャラクター（女性プリセット・丸顔・ショートヘア等）が表示される。
  - キャラクターは地面フィールドの中央付近（原点）に立っている。
  - 右側にカスタマイズパネル（性別/体形/肌の色/目/鼻/口/輪郭/髪/操作の9セクション）が表示され、各コントロールにデフォルト値がセットされている。
  - フッターに「WASDキーまたは矢印キーでキャラクターを移動できます。」の操作説明が表示される。
  - ブラウザのDevTools ConsoleにJSエラーが出ていないこと。
- 実施方法: 手動（目視）。補助的に Console で `document.querySelectorAll('.panel-section').length === 9` などで存在確認可。

### B. 機能テスト（各カスタマイズ項目の反映）

各ケース共通の確認観点: (1) 操作直後に3Dモデルへ即時反映される（リアルタイムプレビュー）、(2) 保存状態表示が「未保存の変更があります」に変わる、(3) 他のパーツ・パラメータに意図しない副作用が出ない。

#### B-1 性別変更 ★★

- 手順: 性別セレクトを「女性」→「男性」に変更する。
- 期待結果:
  - 体形（身長0.95→1.05相当、肩幅0.9→1.1、体重感0.95→1.05）がmaleプリセット値に切り替わり、胴体が太く/大きくなる。
  - 頭・腕・脚の接続位置（`layoutParts()`）が新しいtorso寸法に合わせて再計算され、パーツ間の接続に隙間や埋没が生じない。
  - ※体形スライダーのUI表示値の追従は C-6（重点項目）で別途詳細確認する。

#### B-2 体形（身長）変更 ★

- 手順: 身長スライダーを最小値0.85、最大値1.15、中間値へ動かす。
- 期待結果: `bodyGroup`全体が縦方向にのみスケールし、頭部・胴体・腕・脚が一体で伸縮する（横方向の歪みが無い）。他のパーツの相対位置関係（頭が胴体の上に乗っている等）が崩れない。

#### B-3 体形（肩幅）変更 ★

- 手順: 肩幅スライダーを最小値0.8、最大値1.2へ動かす。
- 期待結果: 胴体の太さ（radius）が変化し、左右の腕の付け根x座標（`armX = 0.28*shoulderWidth+0.09`）が追従して腕が胴体から離れすぎたり埋没したりしない。

#### B-4 体形（体重感）変更 ★

- 手順: 体重感スライダーを最小値0.85、最大値1.25へ動かす。
- 期待結果: 胴体の太さ・長さの両方が変化し、`layoutParts()`により頭・脚の接続y座標が再計算されて、体重感を変えても頭が浮いたり脚が埋まったりしない。

#### B-5 肌の色変更 ★★

- 手順: 肌の色ピッカーをデフォルト`#f2c9a1`から任意の色（例: `#8844cc`）に変更する。
- 期待結果:
  - 頭部の色が変わる。
  - 頬Box（`cheekLeft`/`cheekRight`。skinMaterialを共有参照しているため、square輪郭時に表示すると）も同じ色に変わる。
  - 鼻（`nose`。同じくskinMaterial参照）も同じ色に変わる。
  - 左右の腕（別マテリアルインスタンスだが同色に設定される実装）も同じ色に変わる。
  - 胴体（服）・脚（ズボン）の色は変化しない（固定色仕様）。
- 補足（実装確認事項）: `nose`と`cheekLeft`/`cheekRight`は`head`と同一の`MeshStandardMaterial`インスタンスを共有しているため、`skinColor`のcase処理では`head.material.color`のみ更新すれば連動して変わる実装になっている。テスト実施者はこの3パーツの色が揃って変わることを必ず確認すること（腕は別インスタンスのため個別更新コードが存在し、揃っていることも確認する）。

#### B-6 目（大きさ・間隔・色）変更 ★

- 手順: 目の大きさスライダーを最小0.7・最大1.4に、間隔スライダーを最小0.8・最大1.3に、色ピッカーを任意色に変更する。
- 期待結果:
  - 大きさ変更で左右の目が均等にスケールする。
  - 間隔変更で左右の目のx座標が対称に（`±0.11*spacing`）広がる/狭まる。
  - 色変更で左右の目の色が同時に変わる。
  - 目の変更は鼻・口・輪郭に影響しない。

#### B-7 鼻（大きさ・形状）変更 ★

- 手順: 鼻の大きさスライダーを最小0.7・最大1.4に変更。鼻の形をnormal→small→wideの順に切り替える。
- 期待結果:
  - 大きさ変更で鼻が一様スケールする。
  - 形状変更でジオメトリが差し替わる（normal=円錐大め、small=球、wide=円錐で幅広扁平）。差し替え後も位置（`(0,0,0.31)`、faceFeaturesGroupローカル座標）がずれない。
  - コンソールエラーが出ない（`replaceGeometry`での旧ジオメトリdispose処理を含む）。

#### B-8 口（大きさ・形状）変更 ★

- 手順: 口の大きさスライダーを最小0.7・最大1.4に変更。口の形をnormal→smile→flatの順に切り替える。
- 期待結果:
  - 大きさ変更で口の横幅（scale.x）のみ変わる。
  - `smile`選択時のみ口角が上がったように見える回転（`rotation.x=0.3`）が付く。
  - `normal`と`flat`は同一の見た目（回転無しの横一文字）になる。**これは詳細設計4.2節・13章7点に明記された意図的な仕様であり、不具合ではない。本ケースの結果が「見分けが付かない」であっても不具合報告は行わないこと（H章参照）。**

#### B-9 輪郭変更（基本反映確認） ★★

- 手順: 輪郭セレクトをround→oval→squareの順に切り替える。
- 期待結果:
  - round: 頭部がほぼ真球。
  - oval: 頭部が縦長（`scale(0.9,1.15,0.95)`）。
  - square: 頭部がやや横幅広・縦つぶれ（`scale(1.05,0.95,1.0)`）で、頬Boxが表示される。
  - パーツ位置の崩れ有無・頬Box表示切替の詳細検証はC章で重点実施する（本ケースは基本反映のみのスモーク確認）。

#### B-10 髪型・髪色変更 ★

- 手順: 髪型をshort→long→baldの順に切り替える。髪色ピッカーを任意色に変更する。
- 期待結果:
  - short: 頭部を覆う半球状の髪。
  - long: 半球状の髪に加え、後頭部に垂れる髪（カプセル形状）が追加表示される。
  - bald: 髪が非表示になる（`hairGroup.visible=false`）。
  - 髪色変更は現在の髪型の全パーツに反映される（bald時は見た目に変化なしでも内部エラーが出ないこと）。
  - 髪は頭部の輪郭（scale）に追従して一緒に変形する（headの子であるため。C-1で輪郭と組み合わせて確認）。

### C. 表示・見た目の整合性テスト（重点項目）

#### C-1 輪郭変更時の顔パーツ位置の崩れ確認 ★★★

- 目的: `face.shape` 変更時、目・鼻・口が `faceFeaturesGroup`（headと兄弟・scale固定）に属するため、headの非一様scaleの影響を受けず、常に同じ絶対位置に描画され続けることを確認する（詳細設計4.2.1節、レビュー1〜2回目の重要指摘対応箇所）。
- 手順:
  1. デフォルト（round）の状態で、目・鼻・口の位置関係（顔の中心に対する左右対称性、鼻の高さ、口の高さ）をスクリーンショットまたは目視で記録する。
  2. 輪郭をovalに変更する。
  3. 頭部が縦長になったことを確認しつつ、目・鼻・口が「顔の造作として自然な位置」を保っているか（縦に間延びしたり、顔の輪郭からはみ出したり埋没したりしていないか）を確認する。
  4. 輪郭をsquareに変更する。同様に目・鼻・口の位置崩れが無いか確認する。
  5. 輪郭をroundに戻し、1.の状態と一致することを確認する。
- 期待結果:
  - oval・square何れの場合も、目・鼻・口はhead scale変更の影響を受けず、round時と同じ絶対座標（`faceFeaturesGroup`のローカル座標基準）に留まる。
  - 目・鼻・口が頭部球からはみ出す、めり込む、極端にずれるといった破綻が無い。
  - 輪郭をround→oval→square→roundと一巡させても、目・鼻・口の位置が最初の状態と完全に一致する（scaleが非連動のため、往復操作で誤差が蓄積しないこと）。
  - 髪だけは頭部scaleに追従して伸縮してよい（髪は仕様上headの子でscale継承するため。目・鼻・口とは挙動が異なる点を混同しないよう注意）。
- 実施方法: 手動（目視・スクリーンショット比較）が基本。補足でConsoleから内部座標を検証したい場合、three.jsオブジェクトがグローバル公開されていないため、開発者が一時的に`window`へcharacterを退避するデバッグ改修をしない限りコンソール検証はできない点に注意（現状のmain.jsはcharacterをモジュールスコープに閉じているため）。

#### C-2 square輪郭時の頬Box表示/非表示切替 ★★★

- 目的: 頬Box（`cheekLeft`/`cheekRight`）が`square`選択時のみ表示され、`round`/`oval`では非表示になることを、常駐生成+visible切替方式（詳細設計4.2.2節）に沿って確認する。
- 手順:
  1. デフォルト（round）状態で顔の頬周りを確認する（Boxが見えないこと）。
  2. 輪郭をovalに変更し、頬周りを確認する（Boxが見えないこと）。
  3. 輪郭をsquareに変更し、頬周りを確認する（左右対称にBox形状の頬が見えること）。
  4. squareのまま、目・鼻・口の大きさ/色等、頬に無関係なパラメータを変更し、頬Boxの表示位置・表示状態に影響が出ないことを確認する。
  5. 輪郭をsquare→round、square→ovalへ変更し、頬Boxが非表示に戻ることを確認する。
  6. round⇔oval⇔square を複数回往復し、都度Boxの表示/非表示が正しく切り替わること（表示が残留する、逆に切り替わらない等の不具合が無いこと）を確認する。
- 期待結果:
  - `square`選択時のみ`cheekLeft.visible === true`かつ`cheekRight.visible === true`相当の見た目になる。
  - `round`/`oval`選択時は頬Boxが一切見えない。
  - 頬Boxは肌色（`skinColor`）と同化した色で表示され、顔の輪郭と不自然に浮いて見えない（B-5の肌色変更と組み合わせた確認も行う: squareのまま肌色を変更し、頬Boxの色も追従することを確認）。
  - 往復切り替えを繰り返してもコンソールエラーが出ない・表示状態の残留や消失が無い。
- 実施方法: 手動（目視）。

#### C-3 輪郭×体形の組み合わせ確認（デグレ観点） ★★

- 目的: 輪郭変更（head/faceFeaturesGroup/cheekGroupのscale・visible）と体形変更（layoutParts経由のposition.y再計算）が独立して正しく動作し、互いに干渉しないことを確認する。
- 手順: 輪郭をsquareにした状態で、体重感スライダーを最小・最大に動かす（`layoutParts()`が呼ばれ`head`/`faceFeaturesGroup`/`cheekGroup`の`position.y`が再計算される経路）。
- 期待結果: 体重感変更後も頬Boxが正しい位置（headと同じy座標）に追従し、頭からずれたり浮いたりしない。目・鼻・口も同様にheadに追従する。

#### C-4 輪郭×髪型の組み合わせ確認（デグレ観点） ★

- 手順: 輪郭をoval、髪型をlongにした状態で表示を確認する。
- 期待結果: 髪はheadのscaleに追従して縦長に変形するが、不自然に破綻（頭から大きく浮く、めり込みすぎる等）していないこと。目・鼻・口は非連動のため独立して正しい位置を保つ。

#### C-5 性別変更時の体形スライダー表示追従（重点項目・修正済みバグの再発防止） ★★★

- 目的: 実装後コードレビューで発見・修正された「性別変更後に体形スライダーの表示値が古いまま残る」不具合（詳細設計10.3節・14.4節）が再発していないことを確認する。修正内容は「`onChange`内で`changedPath`による分岐をせず、常に`uiHandle.refreshUI(現在のparams)`を呼ぶ」というもの。
- 手順:
  1. 初期状態（女性、身長1.0/肩幅1.0/体重感1.0）から、身長スライダーを1.10、肩幅を1.05、体重感を0.90に手動でずらす（性別変更前にユーザーが独自にカスタマイズしていたケースを再現）。
  2. 性別セレクトを「女性」→「男性」に変更する。
  3. 3Dモデルの体形が男性プリセット（`height:1.05, shoulderWidth:1.1, weight:1.05`）になっていることを確認する。
  4. カスタマイズパネルの身長・肩幅・体重感スライダーの**表示値**を確認する。
- 期待結果:
  - 3Dモデルの体形は男性プリセット値（1.05 / 1.1 / 1.05）に切り替わる。
  - 手順4で確認する各スライダーの表示値も、モデルと同じ男性プリセット値（1.05 / 1.1 / 1.05）に**追従して更新されている**こと。手順1でユーザーが設定した1.10/1.05/0.90のまま表示が残っていた場合はバグ（修正が退行している）と判断する。
  - さらに、性別を「男性」→「女性」に戻し、スライダー表示がfemaleプリセット（0.95/0.9/0.95）に追従することも確認する。
  - 表示追従後、追従した表示値のまま体形スライダーを操作した場合、意図した値（表示されている値からの変化）でモデルが更新されること（表示値とモデル内部値の不一致が起きていないこと）も併せて確認する。
- 実施方法: 手動が基本。Consoleでの効率化として、各`<input type="range">`要素の`.value`をまとめて取得して比較する簡易チェックが可能。
  ```js
  // DevTools Console で実行（性別変更直後に）
  const rows = [...document.querySelectorAll('#customization-panel .control-row')];
  // 体形セクションの3つのスライダーのvalueを確認（DOM構造に依存するため目視併用推奨）
  document.querySelectorAll('#customization-panel input[type=range]').forEach(el => console.log(el.value));
  ```
  上記は補助的な確認であり、どのスライダーが身長/肩幅/体重感かはUIラベル（テキスト）と合わせて目視で対応付けること（`input`要素自体に`path`属性が付与されていないため、コンソールのみでの完全自動判定はできない点に注意）。

#### C-6 性別変更時の顔パーツ・肌色・髪等の非上書き確認 ★★

- 目的: 詳細設計3.3節「顔パーツ・肌色・髪色などユーザーがカスタマイズ済みの値は保持する」仕様の確認（性別変更でbody系以外が意図せず変わらないこと）。
- 手順:
  1. 肌の色、目の色、髪色をそれぞれデフォルトから変更する。
  2. 性別を変更する。
- 期待結果: body系（身長・肩幅・体重感）のみプリセット値に変わり、肌色・目の色・髪色・輪郭・鼻/口の形状等は手順1で設定した値のまま維持される。

### D. 操作性テスト（移動・カメラ）

#### D-1 WASDキーでの移動 ★★

- 手順: W/A/S/Dキーをそれぞれ単独で押し続け、キャラクターが前後左右（ワールド座標: 前進=-Z, 後退=+Z, 左=-X, 右=+X）に移動することを確認する。
- 期待結果: キー押下中はキャラクターが該当方向へ移動し、離すと停止する。

#### D-2 矢印キーでの移動 ★★

- 手順: ↑↓←→キーでD-1と同様の移動確認を行う。
- 期待結果: WASDと同じ移動方向・挙動になる（`KEY_MAP`でW/↑, S/↓, A/←, D/→がそれぞれ同じ論理入力にマップされているため）。

#### D-3 斜め移動時の速度正規化 ★

- 手順: WとDなど2方向のキーを同時押しして斜め移動させる。
- 期待結果: 斜め移動時に直線移動より速くならない（入力ベクトルを正規化してから`MOVE_SPEED`を乗算する実装のため、体感速度が単一方向移動と同程度であること）。

#### D-4 フィールド範囲内でのクランプ ★★

- 手順: 一方向のキーを長時間押し続け、キャラクターをフィールド端（`FIELD_HALF_SIZE = 4.8`）まで移動させ、さらに押し続ける。
- 期待結果: キャラクターの座標が`-4.8`〜`4.8`の範囲でクランプされ、地面（`PlaneGeometry(10,10)`相当の範囲）の外へ飛び出さない。X軸・Z軸それぞれ正負両方向で確認する（4方向×端到達、計4パターン）。

#### D-5 キャラクターの向き変更 ★★

- 手順: 移動方向を変えながら（例: 前進→右移動→後退）キャラクターの向きの変化を観察する。
- 期待結果: キャラクターの正面（顔のある側）が移動方向へ滑らかに（瞬時ではなく補間的に）追従して回転する。キー入力が無い間は直前の向きを保持する（勝手に回転しない）。

#### D-6 カメラの追従動作 ★★

- 手順: キャラクターを前後左右に移動させ、方向転換も行いながらカメラの動きを観察する。
- 期待結果:
  - カメラはキャラクターの背後やや上方から追従し続け、常にキャラクターが画面内に収まる。
  - キャラクターの向き変更に応じてカメラも回り込むように追従する（背後固定視点）。
  - カメラの追従・注視点変化はなめらか（線形補間により急激なジャンプが起きない）。
  - キャラクターが停止した状態でも、直前の移動慣性でカメラが緩やかに収束し、最終的に静止する。

#### D-7 移動操作とカスタマイズ操作の同時実施（デグレ観点） ★

- 手順: キャラクターを移動させながら、カスタマイズパネルで体形や輪郭を変更する。
- 期待結果: 移動中にパラメータ変更してもエラーが出ず、パーツの再配置（`layoutParts`等）とキャラクターの移動・向き制御が競合せず両立する。

### E. データ永続化テスト

#### E-1 保存ボタン押下→リロード後の復元 ★★

- 手順:
  1. 性別・体形・肌色・目・鼻・口・輪郭・髪型/髪色を全てデフォルトから変更する。
  2. 「保存」ボタンを押す。
  3. 保存状態表示が「保存しました（HH:MM:SS）」になることを確認する。
  4. ページをリロードする（`localStorage`はクリアしない）。
- 期待結果:
  - リロード後、3Dモデルおよびカスタマイズパネルの表示値が、保存時に設定した値で復元される。

  - キャラクターの位置は保存対象外のため、フィールド上の初期位置（原点付近）からスタートする（基本設計5章「位置・向きなどの動的状態は保存しない」仕様の確認）。

#### E-2 パラメータ変更のみ・保存ボタン未押下でのリロード（保存タイミングの確認） ★★

- 手順:
  1. 何らかのパラメータを変更する（保存ボタンは押さない）。
  2. 保存状態表示が「未保存の変更があります」になることを確認する。
  3. ページをリロードする。
- 期待結果: リロード後は変更前の状態（前回保存済みの値、または保存データが無ければデフォルト）に戻る。パラメータ変更操作自体は自動保存をトリガーしないこと（詳細設計8.2節の確認）。

#### E-3 リセットボタン押下 - confirm確認 ★★

- 手順: いくつかパラメータを変更した後「リセット」ボタンを押す。
- 期待結果: `confirm()`ダイアログ（「カスタマイズ内容をリセットします。よろしいですか？」相当の文言）が表示される。

#### E-4 リセットボタン押下 - キャンセル時は変更しない ★

- 手順: E-3のダイアログで「キャンセル」を選択する。
- 期待結果: パラメータ・3Dモデル・UI表示・保存状態表示・localStorageの内容、いずれも変化しない。

#### E-5 リセットボタン押下 - OK時のデフォルト復帰とlocalStorage削除 ★★★

- 手順:
  1. パラメータを変更し「保存」ボタンを押す（localStorageにデータが存在する状態を作る）。
  2. さらにパラメータを追加変更する。
  3. 「リセット」ボタンを押し、confirmダイアログでOKを選択する。
- 期待結果:
  - 3Dモデルの見た目がデフォルトパラメータの状態に戻る。
  - カスタマイズパネルの全コントロール表示値がデフォルト値に戻る（`refreshUI`によるUI再同期）。
  - 保存状態表示が「未保存の変更があります」になる。
  - `localStorage.getItem('3d-character-creator:params')`が`null`になる（`clearStorage()`実行の確認。Consoleで直接確認可能）。
  - リセット後、何もせずページをリロードした場合、保存データが削除済みのため復元されず、デフォルトパラメータで起動し保存状態表示が「デフォルト設定で開始しました」になること（リセットの意図とリロード後動作の一貫性の確認。詳細設計9章の理由説明の裏付け）。

### F. エラー処理テスト

#### F-1 localStorageに保存データが存在しない場合 ★★

- 手順: `localStorage`をクリアした状態でページを開く。
- 期待結果: デフォルトパラメータで起動し、保存状態表示が「デフォルト設定で開始しました」になる。エラーは出ない。

#### F-2 localStorageに壊れたJSON文字列が入っている場合 ★★

- 手順: DevTools Consoleで `localStorage.setItem('3d-character-creator:params', '{not valid json')` を実行後、ページをリロードする。
- 期待結果: `JSON.parse`失敗を`storage.js`が`catch`し、例外が外部に伝播せず、デフォルトパラメータで正常起動する（`restored:false`）。コンソールにアプリを止めるような未処理例外が出ないこと。

#### F-3 localStorageに構造が不正なJSON（一部欠損・型不一致）が入っている場合 ★★

- 手順: 例えば以下をConsoleで設定してリロードする。
  ```js
  localStorage.setItem('3d-character-creator:params', JSON.stringify({
    gender: 'unknown_value',
    body: { height: 999, shoulderWidth: 'abc' },
    skinColor: 'not-a-color',
    face: { shape: 'triangle', eyes: { size: -5 } },
  }));
  ```
- 期待結果:
  - `normalizeParams()`により、`gender`は不正値のためデフォルト`female`に、`body.height`は範囲外(999)のため最大値1.15にクランプ、`shoulderWidth`は数値でないためデフォルト1.0、`skinColor`は不正フォーマットのためデフォルト`#f2c9a1`、`face.shape`は未知列挙値のためデフォルト`round`、`eyes.size`は範囲外(-5)のため最小値0.7にクランプされる。
  - 欠損している`nose`/`mouth`/`hair`等のフィールドも全てデフォルト値で補完され、アプリがエラー無く起動する。
  - `restored:true`扱いとなり保存状態表示は「保存済みのキャラクターを復元しました」になる（部分的にでも復元できるものは復元する方針。8.4節）。
- 実施方法: 手動+Console併用。正規化結果の値そのものはUI表示値・3Dモデルの見た目から間接的に確認する（内部の`normalizeParams`戻り値を直接ログ出力したい場合は一時的なデバッグコード追加が必要な点に留意）。

#### F-4 localStorage自体が利用不可な場合（例: 容量制限、プライベートブラウジング等での例外） ★

- 手順: 可能であればブラウザのプライベート/InPrivateウィンドウ+ストレージ制限設定、または DevTools で `localStorage` へのアクセスが例外を投げるようスタブする（例: Consoleで`Object.defineProperty(window, 'localStorage', { get() { throw new Error('blocked'); } })`のような一時的な検証）などにより擬似的に利用不可状態を作る。
- 期待結果:
  - 読込時: 例外を`loadFromStorage`が`catch`し、デフォルトパラメータで起動する。
  - 保存時: 「保存」ボタン押下で`saveToStorage`が`{ok:false, error}`を返し、保存状態表示が「保存に失敗しました（ブラウザのストレージ設定をご確認ください）」になる。3D表示・カスタマイズ操作自体は継続して利用できる。
- 実施方法: 手動（環境準備がやや特殊なため優先度は中程度。実施できない場合はコードレビューでの`try/catch`網羅確認で代替する旨を記録する）。

#### F-5 WebGL非対応環境でのエラーメッセージ表示 ★★

- 手順: WebGLを無効化した環境（例: Chromeで`chrome://flags`のWebGL無効化、またはGPUプロセスを`--disable-webgl`起動オプション等で無効化）でページを開く。あるいはDevTools Consoleで`HTMLCanvasElement.prototype.getContext`を一時的に`webgl`/`webgl2`について`null`を返すよう上書きしてから`init()`相当の動作を再現する。
- 期待結果:
  - 3Dキャンバス領域に「お使いのブラウザ/環境では3D表示に対応していません。最新のChromeまたはEdgeでお試しください。」というメッセージが表示される。
  - カスタマイズパネルは初期化されない（操作対象が無いため。`isWebGLAvailable()`が`false`の場合`init()`はその時点で`return`し、以降の処理を一切行わない実装のため、パネルへのコントロール生成やイベント登録も発生しない）。
  - コンソールに`console.error('WebGL is not available in this environment.')`のログが出力される。
  - JS例外でアプリ全体がクラッシュしたような状態（白画面、他のUIも含めて完全に機能しない等）にはならない（ヘッダー・フッターの静的HTMLは表示されたままである）。
- 実施方法: 手動（環境準備がやや特殊）。簡易代替として、DevTools Consoleで以下を試すことでロジック単体の妥当性を間接確認できる。
  ```js
  // 現在のブラウザの判定結果を確認する参考コード（実装同等ロジック）
  (() => {
    try {
      const c = document.createElement('canvas');
      return !!(c.getContext('webgl2') || c.getContext('webgl'));
    } catch { return false; }
  })();
  ```

### H. 仕様確認（意図的な仕様の再確認・誤報告防止）

#### H-1 口の形状 normal / flat の見た目同一性は仕様通りであることの確認 ★★

- 目的: B-8で確認した「`normal`と`flat`が同一の見た目になる」事象について、テスト実施者が誤ってバグとして起票しないよう、仕様であることを明文化して確認する。

- 注意事項: 本ケースは「バグを探す」テストではなく「意図した仕様からの逸脱が無いか」を確認するテストである。テスト結果報告書には「仕様通り（正常）」として記録し、不具合起票は行わないこと。ただし、`normal`と`flat`以外の組み合わせ（例: `smile`）まで区別が付かなくなっていた場合はデグレであり、その場合は不具合として報告する。

## 4. 実施方法まとめ

- 主体: ブラウザでの手動確認（Chrome/Edge最新版）。
- Consoleでの補助確認が可能な項目: A-2（表示値確認）, C-5（スライダー値の一括取得）, E-5（`localStorage.getItem`によるデータ削除確認）, F-1〜F-3（`localStorage.setItem`による疑似データ投入）, F-5（WebGL可否判定ロジックの単体確認）。
- Consoleのみでは完結せず必ず目視が必要な項目: C-1, C-2, C-3, C-4, B系の見た目反映確認全般, D系（移動・カメラ）, G系（リサイズ）。3Dオブジェクトの内部状態（`character.parts`等）は`main.js`のモジュールスコープに閉じておりグローバル公開されていないため、位置・scaleの数値をConsoleから直接検証することはできない（目視・スクリーンショット比較が前提）。
- 自動テスト化の余地: `params.js`の`normalizeParams`/`createDefaultParams`/`getGenderBodyPreset`はDOM非依存の純粋関数であるため、将来的にNode.js上でのユニットテスト（例: `node --experimental-vm-modules`やVitest等の導入）に切り出しやすい。F-2〜F-4・I-4の一部はこの単体関数テストで代替・補強可能だが、本MVPの現状構成にはテストランナーが導入されていないため、本仕様書では手動+DevTools Console確認を正式な実施方法とする。

## 5. 不具合報告時の注意事項（再掲）

- 口の形状`normal`と`flat`が同一の見た目になる事象は**仕様**である（H-1参照）。この点のみを理由とした不具合報告は行わないこと。
- それ以外の「設計書に明記された挙動との差異」を発見した場合は通常通り不具合として報告する。特にC-1・C-2・C-5は過去に複数回の設計修正・コードレビュー指摘が入った箇所であるため、期待結果との差異を発見した場合は優先度を高めに報告すること。
