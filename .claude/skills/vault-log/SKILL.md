# Vaultへの作業ログ記録

このスキルが呼ばれたら、秘書（メインの会話）は以下の手順で進める。サブエージェント（obsidian-scribe）は会話の記憶を持たないため、要約と構成は必ず秘書自身が行い、書くべき内容を全文プロンプトに書き下してから依頼すること。

## 1. 内容の要約（秘書が行う）

この会話の中で実際に何をしたかを、以下の観点で整理する。
- 背景・きっかけ（何を頼まれた/何が起きたか）
- やったこと・原因調査・対処内容（あれば手順や発見した原因も具体的に）
- 結果・結論
- 教訓・次回への申し送り（あれば）

雑談や試行錯誤の全部を書き写す必要はない。後で読んで要点が分かるレベルに整理する。

## 2. 保存先とタグの決定

`SecondBrain/運用ルール.md` を（`obsidian-scribe`経由、または既にMCPツールが手元にあればそれで）確認し、フォルダ構成・命名規則・タグ運用ルールに従う。基本方針：
- 作業ログ・日次的な記録 → `SecondBrain/30_Resources/YYYY-MM-DD <内容が分かるタイトル>.md`（YYYY-MM-DDは実行日）
- 進行中プロジェクトの一環なら `SecondBrain/10_Projects/` 配下も検討
- 既存の関連ノートがあれば `[[ダブルブラケット]]` でリンクする
- 横断的なテーマはタグで管理する（例: `#claudecode` `#mcp` `#トラブルシューティング`）

**注意**: `運用ルール.md` 内のフォルダ表記（`30_Resources/` 等）はVault内のノートから見た相対表記であり、MCPツールの `filepath` はVaultルート基準。Vaultルート直下は `SecondBrain/` フォルダそのものなので、`obsidian-scribe`へ依頼するパスは必ず `SecondBrain/` から書き始めること（`SecondBrain/`を付け忘れるとVaultルート直下に同名フォルダが誤って新規作成されるので要注意）。

## 3. obsidian-scribeへ依頼

`obsidian-scribe` に、1で整理した内容をそのまま（省略せず）本文として渡し、2で決めたファイルパス・タグ・リンクを指定してノート作成を依頼する。

## 4. 検証

サブエージェントの完了報告は鵜呑みにせず、可能であれば秘書自身がMCPツール（`mcp__obsidian__obsidian_get_file_contents` 等）でノートが実際に作成され、内容が指示通りかを直接確認する。確認できない場合はその旨を正直にユーザーに伝える。

## 5. 報告

保存先のファイルパスと記録した内容の要点を、簡潔にユーザーへ報告する。
